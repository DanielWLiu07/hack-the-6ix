# Lidar review notes — findings, hardcodedness, decisions

Written 2026-07-18 during a code review of the lidar + dashboard path, hotspot
off (no hardware). Companion to [README.md](README.md). This is the "what I found
and what you need to decide" doc; the README is the "how it works" reference.

---

## Why the dashboard "didn't look right"

It was being fed the wrong source. The preview ran the **Python `sim.py`**, which
emits **only `lidar_scan`** — no `slam_pose`, no `slam_map`. But the `/lidar` page
is built around the SLAM view (green accumulating map + amber pose marker), so
with a lidar-only source you get the bare **robot-centered radar fallback** and
none of what the page copy promises. On top of that the live points came from
`sim.py`'s **8×6 m** room while the 3D backdrop `world.glb` is a **4×3 m**
placeholder, so the cloud spilled outside the backdrop.

**Fix for previewing:** open `http://localhost:5173/lidar?sim=1`. The browser sim
(`web/src/lib/sim.js`) generates the full stream client-side — map, pose, and
sweep all coherent, no backend, no hotspot. See README → "Previewing without
hardware." Nothing about the renderer is broken.

## Hardcodedness inventory

Sorted by how likely it is to bite. None of these are bugs on their own; they're
placeholders/assumptions to be aware of and to revisit before the demo.

1. **`world.glb` is a synthetic placeholder, not a real scan.**
   Generated by `phone/make_sample.py` — a 4×3×2.5 m vertex-colored fake room.
   Until the real iPhone scan is run through `phone/process.py` (same output
   path), the 3D backdrop is fake and won't match whatever room you're actually
   in or whatever the C1 sees. **Action:** capture the real room and re-run
   `process.py` at the venue.

2. **Three different hardcoded rooms.** Nothing enforces they agree:
   - `web/server/sim.js` (node): **6×4 m**, 180-pt scans.
   - `robot/lidar/sim/sim.py` (python): **8×6 m**, 360-pt scans.
   - `phone/make_sample.py` → `world.glb`: **4×3 m** backdrop.
   With the real robot the live scan is the real room, so this only matters for
   sim previews — but it's why sim + backdrop never line up. **Decision:** pick
   one canonical sim room and match `world.glb`'s placeholder to it, or just
   always preview with `?sim=1` (whose room + map are internally consistent).

3. **Fixed camera, no auto-fit on the page.** `LidarView.jsx` passes
   `camera={position:[0,4.5,4.5], fov:55}` and leaves `autoFitEnabled=false`,
   `fitView=false`. The view is framed for a ~4–6 m room; a much larger/smaller
   real room won't be framed well until you orbit/zoom. **Decision:** consider
   enabling `fitView`/`autoFitEnabled` for the live page, or leave manual (the
   auto-fit has its own edge cases — see the long comments in `WorldModel`).

4. **Render constants** in `LidarViewport.jsx` (fine, just know where they are):
   `SENSOR_H=0.15` (scan/marker height off the floor), `DECAY_MS=4000`,
   `MAX_SCANS=16`, point color `#46e068`, size `0.05`, cream background
   `#f4f0e6`, SLAM colors. The page overrides only camera + controls.

5. **`ANGLE_OFFSET_DEG=0`, `ANGLE_CCW=0`** in `pi/lidar_node/config.py` — assumes
   the C1's 0° already points robot-forward and it's mounted right-side up. Almost
   certainly needs tuning once physically mounted (points will be rotated).

## Bugs found

| # | File | Issue | Status |
|---|---|---|---|
| 1 | `pi/lidar_node/emitter.py` | Connected **without** `auth={"role":"robot"}` → hub tagged the Pi as a dashboard and **silently dropped every scan** (`emit` returned True). | **FIXED** (uncommitted) — added auth; verified end-to-end via `LIDAR_MOCK=1` → 23 scans relayed to a ui subscriber. |
| 2 | `pi/lidar_node/emitter.py` | `ts` was `time.time()` (epoch **seconds**); contract + sim + frontend `fmtTime` all use epoch **ms**. | **FIXED** (same change) — converts to ms; verified `ts_digits=13`. |
| 3 | `web/server/sim.js` → `blob.js` | Imports `@vercel/blob` at module top; the package isn't installed locally, so `node sim.js` **and the panic-switch demo fallback** throw `ERR_MODULE_NOT_FOUND` and don't start. | **Open** — low priority. Either `npm i` it in `web/server`, or make `blob.js`'s import lazy so the sim/panic path doesn't need it. The lidar preview doesn't need it (`?sim=1` covers previews). |

Note: `@vercel/blob` **is** listed in `web/server/package.json`, but wasn't in
the installed tree during review — a plain `npm install` in `web/server` may
resolve #3; worth confirming.
